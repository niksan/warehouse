# Дорожная карта: учебный Go-проект (Итерация 1)

> **Режим работы:** гид. Перед каждым срезом Claude объясняет Go-идиому и ставит задачу;
> **код пишешь ты**; Claude ревьюит на идиоматичность и объясняет ошибки. Скучную обвязку
> (Dockerfile, docker-compose, конфиг sqlc, шаблоны миграций) генерирует Claude — на ней учиться нечему.
>
> **Горизонт:** ~2 недели, ~2 ч/день ≈ 20–24 ч. Сессия = один рабочий заход (~2 ч).
> Основа — ТЗ `go-learning-project-iteration-1.md`. Порядок изменён: **логика на памяти сначала, БД и Docker в конце.**
>
> **Ритуал каждой сессии:** `go build ./...` → `go vet ./...` → `go test ./... -race`. Зелёное — коммит.

---

## Фаза 0 — Каркас и тулчейн (0.5 сессии)

- [ ] **С0.** Переименовать модуль `hello_world` → нормальное имя (`go mod edit -module ...`).
      Создать структуру пакетов из §3 ТЗ (`cmd/app`, `internal/...`, `migrations`, `deploy`).
      Прогнать `go build`, `go test`, `go vet` — прочувствовать тулчейн (аналог `bundle`/`rake`).
  - *Идиомы:* модуль vs пакет, `internal/`, `cmd/`, layout Go-проекта.
  - *Артефакт:* пустой скелет, который собирается.

## Фаза A — Тонкий HTTP-срез, БЕЗ БД (2 сессии)

- [ ] **С1.** `net/http`: роутинг по методу+пути (Go 1.22), один хендлер, отдающий JSON.
      Структуры запроса/ответа с тегами `json`. Запустить сервер, дёрнуть `curl`.
  - *Идиомы:* `http.Handler`/`HandlerFunc`, структуры, JSON-теги, `json.Marshal`, value vs pointer receiver.
  - *Артефакт:* сервер отвечает на `GET /health` и заглушку `GET /orders`.
  - *Rails-мостик:* хендлер ≈ контроллер, но без магии — маршрут регистрируешь руками.

- [ ] **С2.** `slog`-middleware (метод, путь, статус, длительность, request-id). `context` — прокинуть request-id.
      Первый **table-driven** тест на хендлер + `httptest`. Прогнать `go test -race`.
  - *Идиомы:* `log/slog`, middleware как `func(http.Handler) http.Handler`, `context.Context`, `httptest`, table-driven тесты.
  - *Артефакт:* каждый запрос логируется структурно; тест зелёный под `-race`.
  - *Фраза для собеса:* почему middleware в Go — это обёртка над `http.Handler`, а не «фильтр» как в Rails.

## Фаза B — `inventory`: сердце проекта, мьютекс (3 сессии)

- [ ] **С3.** Доменные типы склада. In-memory store `map[warehouseID]map[sku]int`. Метод `GetStock`.
  - *Идиомы:* maps, вложенные maps, методы на структуре, нулевые значения, инкапсуляция через приватные поля.
  - *Артефакт:* store, засеянный хардкодом (Postgres позже), `GetStock` отдаёт снапшот.

- [ ] **С4.** `Reserve` под `sync.Mutex`: атомарный `lock → check → decrement → unlock`.
      Кастомная ошибка `ErrInsufficientStock`, обёртка `%w`, проверка через `errors.Is`.
  - *Идиомы:* `sync.Mutex`, `defer mu.Unlock()`, ошибки как значения, кастомный тип ошибки, `fmt.Errorf("...: %w")`, `errors.Is`.
  - *Артефакт:* `Reserve` корректно отклоняет нехватку стока.
  - *Фраза для собеса:* что такое TOCTOU и почему `check+decrement` обязан быть под одним локом.

- [ ] **С5.** Конкурентный тест: N горутин параллельно дёргают `Reserve`, проверить консистентность
      (сток не ушёл в минус, брони не потерялись). Обязательно `go test -race`.
  - *Идиомы:* `sync.WaitGroup`, запуск горутин в тесте, детектор гонок, детерминизм проверки итога.
  - *Артефакт:* тест, который зелёный под `-race` и краснеет, если убрать мьютекс.
  - *Фраза для собеса:* это ядро проекта — «конкурентность приходит бесплатно от HTTP-сервера» (§2, §8 ТЗ).

## Фаза C — `fulfillment`: синхронный оптимизатор (1–2 сессии)

- [ ] **С6.** Жадный проход по модели стоимости (§6): домашний склад дёшево, прочие — плоский штраф,
      `+ split_penalty` за дробление. Чистая функция `Plan`. Table-driven тесты на кейсы
      (всё из дома / досорсить / нехватка → `feasible=false`).
  - *Идиомы:* слайсы, сортировка, чистые функции без сайд-эффектов, table-driven на бизнес-логику.
  - *Артефакт:* `Plan` считает раскладку и стоимость; тесты покрывают ветки.

## Фаза D — `orders`: слои и интерфейсы-швы (3 сессии)

- [ ] **С7.** Определить `InventoryPort` и `FulfillmentPort` **в пакете `orders`** (интерфейс определяет
      потребитель). Реализации `inventory`/`fulfillment` их удовлетворяют конкретными типами. Связать в `main.go`.
  - *Идиомы:* «accept interfaces, return structs», интерфейс на стороне потребителя, dependency wiring вручную (без DI-контейнера).
  - *Артефакт:* `orders` зависит от двух интерфейсов, не от конкретных пакетов.
  - *Фраза для собеса:* почему в Go интерфейс живёт у потребителя, и как это станет линией распила на сервисы (§3, §12).

- [ ] **С8.** Оркестрация `POST /orders` — 7 шагов из §4 (валидация → GetStock → Plan → Reserve → save → ответ).
      `409 Conflict` на `feasible=false` и на нехватку стока. Обёртки ошибок через слои.
  - *Идиомы:* оркестрация в service-слое, маппинг доменных ошибок в HTTP-статусы, `errors.As`, ранний возврат.
  - *Артефакт:* полный синхронный сценарий заказа (пока сток in-memory, заказ — заглушка в памяти).

- [ ] **С9.** Юнит-тесты `orders`-service с **моками портов** (интерфейсы это позволяют).
      Проверить ветки: успех, `feasible=false`, гонка на `Reserve`.
  - *Идиомы:* моки через реализацию интерфейса вручную (без mock-фреймворка), изоляция слоёв в тестах.
  - *Артефакт:* сервис протестирован без БД и HTTP.
  - *Фраза для собеса:* швы-интерфейсы окупились — сервис тестируется в изоляции.

## Фаза E — Postgres: делаем «по-настоящему» (3 сессии)

> Здесь Claude **генерирует** конфиг `sqlc` и шаблоны миграций (обвязка); ты пишешь SQL-запросы и репозитории.

- [ ] **С10.** Миграции (`goose`/`golang-migrate`) для таблиц из §4/§5. Подключение `pgx` в `platform`.
  - *Идиомы:* пул соединений `pgxpool`, миграции как версионируемый SQL, `context` в запросах.
  - *Артефакт:* БД поднимается, схема накатывается.

- [ ] **С11.** Репозитории `inventory` (сид стока из Postgres при старте, персист при `Reserve`) через `sqlc`.
  - *Идиомы:* `sqlc` (типобезопасный Go из SQL), репозиторий как реализация интерфейса, разделение in-memory кэш vs источник истины.
  - *Артефакт:* сток живёт в памяти под мьютексом, но засеивается и персистится в Postgres.

- [ ] **С12.** Репозиторий `orders`: сохранить `CONFIRMED`-заказ + раскладку (`order_shipments`).
      `GET /orders/{id}` и `GET /orders` из БД.
  - *Идиомы:* транзакция `BEGIN…COMMIT` (заказ + позиции + отгрузки атомарно), маппинг строк в доменные структуры.
  - *Артефакт:* заказы персистятся и читаются из Postgres.
  - *Фраза для собеса:* в монолите транзакция БД закрывает атомарность; в Итерации 2 при распиле она исчезает → нужна сага (§12).

## Фаза F — Инфраструктура и полировка (2 сессии)

- [ ] **С13.** Multi-stage `Dockerfile` + `docker-compose` (app + Postgres, больше ничего). Конфиг через env.
      **Graceful shutdown:** `os/signal` → отмена корневого `context` → `server.Shutdown`, дождаться запросов.
      Опционально: `select` по сигнальному каналу и каналу ошибки сервера (§7 — единственный честный дом каналов в Итерации 1).
  - *Идиомы:* multi-stage сборка, env-конфиг, `signal.NotifyContext`, `http.Server.Shutdown`, (опц.) `select` + каналы.
  - *Артефакт:* `docker compose up` поднимает всё; Ctrl+C гасит сервер без потери запросов.
  - *Фраза для собеса:* конкурентность vs параллелизм (§8) + идиома graceful shutdown.

- [ ] **С14.** Добить тесты по чеклисту §13. Финальный `go test ./... -race`. Короткий README:
      по каждому нетривиальному решению — «почему так» (это твоя шпаргалка к собесу).
  - *Артефакт:* всё зелёное под `-race`; README с обоснованиями решений.

## Как этим пользоваться

1. Начинаешь сессию — говоришь Claude: «делаем С_N». Claude объясняет идиому и ставит задачу.
2. Пишешь код сам. Застрял — спрашивай «почему», а не «напиши за меня».
3. Готово — просишь ревью. Claude показывает неидиоматичные места и объясняет.
4. Зелёный `-race` → отмечаешь галочку здесь → коммит.

**Отставание — это норма.** Если срез не влез в 2 ч — это не провал графика, а сигнал, что тема
глубокая (обычно С4–С5 про мьютекс и С10–С11 про Postgres). Лучше понять, чем успеть.
